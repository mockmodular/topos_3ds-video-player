# 2026-04-02 音频解码线程迁移至C1（Steps 1-6）

## 动机
O3DS C0 占用率100%，偶发卡顿。C1 空闲。目标：音频解码移C1，C0专给视频。

## 实施步骤

### Step 1（vid_state.h）
新增：
- `Vid_command`：`AUDIO_DECODE_THREAD_DECODE_REQUEST / SEEK_REQUEST / ABORT_REQUEST / SHUTDOWN_REQUEST`
- `Vid_notification`：`AUDIO_DECODE_THREAD_FINISHED_SEEKING_NOTIFICATION / FINISHED_ABORTING_NOTIFICATION`
- `Vid_player` 字段：`Thread audio_decode_thread`、`Queue_data audio_decode_thread_command_queue`
- struct `Vid_audio_decoded_data { uint8_t* audio; uint32_t audio_samples; double pos; uint8_t packet_index; }`
- `vid_decode.h` 新增 `void Vid_audio_decode_thread(void* arg)` 声明

### Step 2（vid_lifecycle.c）
- `Vid_init_thread`：`Util_queue_create(audio_decode_thread_command_queue, 200)`
- 临时在 if/else 块外以 core 0 创建 audio_decode_thread（stub）
- `Vid_exit_thread`：join + free audio_decode_thread；queue_delete

### Step 3（vid_decode.c）
- `Vid_decode_thread` 中：音频解码后 malloc `Vid_audio_decoded_data`，`Util_queue_add` 给 audio_decode_thread
- 更新 `last_decoded_audio_pos`
- 去掉 num_of_audio_buffers stall 条件

### Step 4（vid_decode.c）
- `Vid_audio_decode_thread` stub → 完整实现：
  - `Util_converter_convert_audio`（PCM格式转换，最多2ch S16）
  - 音量缩放（int16，溢出clamp INT16_MAX，设 TOO_BIG sub_state）
  - `Util_speaker_add_buffer` 重试125次×2ms
  - free converted / audio / audio_data
- 效果：有声音，不卡死，seek 正常

### Step 5（vid_lifecycle.c）
- `threadCreate(Vid_audio_decode_thread, ..., core=1)`（从0→1）
- 权限：`APT_SetAppCpuTimeLimit` O3DS=70% / N3DS=80% 已在 Vid_init 中申请，无需额外改动
- 效果：C0 占用从100%降至~95%，帧率更稳

### Step 6（vid_decode.c）
- **Seek handler**：`Util_speaker_clear_buffer` 后发 `AUDIO_DECODE_THREAD_SEEK_REQUEST`（SEND_TO_FRONT，**不等通知**）
- **Abort handler**：等 video_thread 后发 `AUDIO_DECODE_THREAD_ABORT_REQUEST`（SEND_TO_FRONT，**等** ABORTING 通知）再 speaker_exit/decoder_close
- **audio_decode_thread SEEK/ABORT case**：drain 队列旧 DECODE_REQUESTs（free，不送 speaker），ABORT 时发通知，SEEK 时不发

## 关键教训：decode_thread 通知队列不可 blocking 等待
`decode_thread_notification_queue` 在主循环末用 timeout=0 非阻塞 poll。
命令 handler 里若 while(true) 等该队列的特定通知 → 吃掉其他线程的通知 → seek 永远卡在 PREPARE_SEEKING。
**规则**：Seek = fire-and-forget；Abort（其他线程已全停）= 可 blocking 等。

## 实测结果
- Step 4：有声音，seek 正常，不卡死
- Step 5：C0 从100%→~95%，帧率提升感受明显
- Step 6（修复前）：seek 卡死（notification 被吃掉）→ 修复为 fire-and-forget → seek 完美
