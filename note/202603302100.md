## NEON 崩溃（第一版 ASM 库上机即死）

### 现象
刷入新 CIA 后播放 MPEG-2 MKV（800×240）立刻黑屏死机。
crash_dump 文件：`crash_dump_00000058~61.dmp`（项目根目录）

### 崩溃根因分析
1. 去掉 `--disable-asm` 后，GNU as 能汇编 NEON 语法（GNU as 支持所有 ARM 指令集语法，不管目标 CPU 是否有 NEON）
2. FFmpeg configure 探测汇编器 NEON 支持 → `HAVE_NEON 1` 写入 config.h
3. arm-none-eabi（非 Linux kernel）→ `ff_get_cpu_flags_arm()` 走 `#else` 分支，直接 `return AV_CPU_FLAG_NEON * HAVE_NEON`
4. `blockdsp_init_neon.c`（编译进库）把 `clear_blocks` 指针指向 `ff_clear_blocks_neon`
5. MPEG-2 解码调用 `clear_blocks` → ARM11 执行 NEON 指令 → **非法指令崩溃**

崩溃地址解析（arm-none-eabi-addr2line）：
- PC = `0x002933F4` → `mpeg_decode_slice`（mpeg12dec.c）
- LR = `0x00602EC8` → `ff_clear_blocks_neon` ← NEON 函数被调用

附注：`HAVE_NEON_EXTERNAL 0` 但 `HAVE_NEON 1`——两个值含义不同：
- `HAVE_NEON_EXTERNAL`：外部 ASM 文件是否编入（0=no）
- `HAVE_NEON`：configure 认为硬件支持 NEON（1=yes，被 GNU as 探测欺骗）
- 非 Linux 分支用的是 `HAVE_NEON`，不是 `HAVE_NEON_EXTERNAL`，所以即使外部 ASM 文件编进去了，分发函数也会被激活

### 修复
`build_ffmpeg_asm.sh` 加 `--disable-neon`，重编后：
- `HAVE_NEON 0` / `HAVE_NEON_EXTERNAL 0`
- libavcodec.a 里零 NEON .o 文件
- 项目编译 0 error 0 warning

## make clean 必须做（教训）

第一次替换 .a 文件后直接 `make`，输出只有：
```
Building 3dsx...
Created banner...
Building cia...
```
没有 `linking` 步骤 → 旧 .elf 被复用，新 ASM 代码根本没链进去。

**每次替换 library/lib/*.a 后必须 `make clean && make`。**

## ASM:A6 运行时指示器（UI 验证手段）

在 `video_player.c` 下屏 DECODE 行加了 `ASM:` 字段：
```c
static int cached_cpu_flags = -1;
if(cached_cpu_flags < 0)
    cached_cpu_flags = av_get_cpu_flags();
const char* asm_tag =
    (cached_cpu_flags & AV_CPU_FLAG_ARMV6)   ? "A6"  :
    (cached_cpu_flags & AV_CPU_FLAG_ARMV5TE) ? "A5"  : "--";
```
DECODE 行格式：`Dec:SW/CPU   Tex:LINEAR  SAR:1/1  ASM:A6`
- `A6` = ARMv6 ASM 已激活（目标状态）
- `--` = 库被换回旧版或配置错误

## 理论加速预期

| 解码内核 | 占 MPEG-2 CPU% | ARMv6 ASM 提升 |
|---|---|---|
| IDCT（simple_idct_armv6） | ~40-50% | ~1.5-2× |
| 运动补偿（hpeldsp_armv6 / mpegvideo_armv5te_s） | ~25-35% | ~1.3-1.6× |

整体估算：解码线程快 30-50%，实际 FPS 提升取决于解码占帧时间比例，预计 **15-35%**。
仅对 FFmpeg 软解路径有效（MPEG-2/MPEG-4 等）；H.264 MVD 硬解不走 FFmpeg，无影响。

## Y2R SBS gap 参数风险（待上机验证）

当前方案：两次 Y2R + SetSendingY/U/V gap 参数跳过右/左半。
- 参数逻辑正确（gap=half_w 跳过另一半），但 Y2RU 输入侧 gap 是否有效**未经硬件验证**。

**风险**：若 Y2RU_SetSendingY gap 被忽略（仅输出侧生效），硬件顺序读 96000B = 前120行×800px，图像变成上半帧垂直拉伸。

回退方案（更可靠）：
1. 一次 Y2R（800×240）→ BLOCK_8_BY_8 768KB
2. CPU tile split：30个tile行 × 2次 × 12800B memcpy_asm = 768KB
3. 两次 `Vid_large_texture_set_data` 上传

tile split 约 2-4ms CPU 时间，比旧的逐像素 memmove 快很多，比双 Y2R 可靠。

## N3DS slice_cores 优化（低优先级待办）

`vid_settings.c:69~73` 当前 N3DS 把 slice_cores 对所有可用核全开（c0/c1/c2/c3）。
c0 是主线程（渲染+HID），c1 是读包线程，FFmpeg slice 线程占用它们会导致竞争。
建议后续限制 slice_cores 仅 c2+c3：
```c
// N3DS分支改为：
slice_cores[2] = Util_is_core_available(2);
slice_cores[3] = Util_is_core_available(3);
// c0/c1不给FFmpeg slice线程
```
