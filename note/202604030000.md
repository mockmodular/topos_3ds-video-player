# 改动记录 2026-04-03

## 本次会话主要变更

### 1. 所有主线程统一迁至 C0（vid_lifecycle.c）
之前 note 中记录线程在 C1 是错误的。当前实态：
- N3DS / O3DS 全部主线程（decode_thread / decode_video_thread / convert_thread / read_packet_thread / audio_decode_thread）**全部在 C0**
- FFmpeg worker 线程（fake_pthread）才是分核的唯一路径

### 2. fake_pthread worker 线程改动
- 优先级：`DEF_THREAD_PRIORITY_HIGH` → `DEF_THREAD_PRIORITY_NORMAL`
- `Util_fake_pthread_set_enabled_core` 新增 `int start_core` 参数（指定从哪个核开始轮询）
- N3DS：list=[C0,C2,C3], start_core=2 → 轮序 C2→C3→C0
- O3DS：list=[C0,C1], start_core=1 → 轮序 C1→C0→C1→C0

### 3. DecoderVideoSoft_set_enabled_cores / Util_decoder_video_set_enabled_cores 新增 worker_start_core 参数
传参链：vid_settings.c → decoder.c → decoder_video_soft.c → fake_pthread

### 4. FFmpeg 线程模式选择逻辑改为 FRAME 优先
- 旧：SLICE 优先（H.264 有 SLICE 能力但 1-slice/frame → C0 独占）
- 新：**FRAME 优先，SLICE 次之**
  - H.264：有 AV_CODEC_CAP_FRAME_THREADS → FF_THREAD_FRAME（workers 解完整帧）
  - H.265：只有 AV_CODEC_CAP_SLICE_THREADS → FF_THREAD_SLICE（CTU 行级并行）
  - MPEG-2：只有 SLICE → FF_THREAD_SLICE（宏块行级，10~30 jobs/frame）
  - AV1/dav1d：两者皆无 → thread_type=0，dav1d 自己管线程池

### 5. receive=EAGAIN bug 修复（decoder_video_soft.c）
send=0（包已接受）但 receive=AVERROR(EAGAIN)（流水线填充中），
旧代码当错误处理；新代码跳 `need_more_input` 标签：消费包，返回
`DEF_ERR_NEED_MORE_INPUT`，调用方取下一包继续填充帧线程流水线。

### 6. Slice_cores 未传给 fake_pthread bug 修复
旧代码：`Util_fake_pthread_set_enabled_core` 调用被注释掉，vid_settings 配好的
核心列表从未真正生效。现在在 avcodec_open2 前无条件调用（按 FRAME/SLICE 分支）。

### 7. vid_settings.c O3DS 分支加入 C1
O3DS worker pool：frame_cores[1]=true, slice_cores[1]=true（之前只有 C0）

## 性能测试结果（用户记录）
- O3DS MPEG-2 800×240 SBS 3D，N3DS 开 O3DS 模式测试：**24fps 稳定**
- thread_count=3 时，O3DS：C0 承载 decode_video_thread + 1 worker（共 2 解码实体），C1：1 worker
- 理论解码性能提升：预计 20%～50%
