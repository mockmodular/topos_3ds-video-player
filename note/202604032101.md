## 1. SEM Advanced HW/SW decoder 按钮三项 Bug 修复

**文件**：`source/system/sem.c`

### Bug 1：反应慢（根本原因）

`sem_hw_decode_hw_button.selected` 和 `sem_hw_decode_sw_button.selected` 未加入 `Util_watch_add`。按下按钮后 selected 变化，但 watch 未触发 → 屏幕不立刻刷新 → 感觉响应慢。

**修复**：在 Advanced watch_add 区域补充：
```c
Util_watch_add(WATCH_HANDLE_SETTINGS_MENU, &sem_hw_decode_hw_button.selected, sizeof(...));
Util_watch_add(WATCH_HANDLE_SETTINGS_MENU, &sem_hw_decode_sw_button.selected, sizeof(...));
```

### Bug 2：视觉"两个框框重叠"

两个按钮 width=140 各自，总共占 290px（320px 屏幕几乎全宽），两大块 DEF_DRAW_WEAK_AQUA 半透明色块紧挨 10px 间隙，视觉上像融合重叠。

**修复**：width 140→130，x=15→10 / x=165→150，结果：10+130+10+130+40=320，右侧留 40px 与其他按钮行风格一致。

### Bug 3：selected 状态永不清除

`sem_hw_decode_hw_button` / `sem_hw_decode_sw_button` 在 DESEL 区域（lines ~1100）**没有** DESEL 处理。一旦按下 selected=true，永远不清除。

**修复**：在 DESEL 区补充：
```c
if(HID_ADVANCED_HW_DECODE_HW_DESEL(*key) || sem_scroll_mode)
    sem_hw_decode_hw_button.selected = false;
if(HID_ADVANCED_HW_DECODE_SW_DESEL(*key) || sem_scroll_mode)
    sem_hw_decode_sw_button.selected = false;
```

### 省电模式机制说明

绘制函数判断条件（sem.c + vid_draw.c 均有）：
```c
if(Util_watch_is_changed(watch_handle_bit) || Draw_is_refresh_needed() || !config.is_eco)
    // 执行绘制
```

| 状态 | 条件 | 效果 |
|------|------|------|
| Eco ON（默认 true） | `!is_eco = false`，只有 watch 变化或显式 refresh 时重绘 | 跟着内容帧率刷新，节省 CPU |
| Eco OFF | `!is_eco = true`，**每帧强制重绘** | 固定 60fps 重绘，浪费 C1 算力 |

**视频播放时推荐 Eco ON**：每解出一帧 watch 触发，刷新与视频帧率精确同步。Eco OFF 会在 C1 每帧多做无效重绘，浪费已紧张的 C1 资源（音频解码 + convert 线程也在 C1）。

**默认值**：`config.is_eco = true`（正确的默认）。

## 3. N3DS 硬解画质分析（无代码改动）

**问题**：硬解画质略差于软解，怀疑是缺少滤镜。

**结论**：不是滤镜问题，是颜色转换路径不同：
- 软解：YUV420p → Y2R 硬件（BT.601 Limited Range）→ RGB
- 硬解：MVD 内部直接输出 ABGR32（MVD 自己的色彩矩阵，略有差异）

纹理线性滤镜（bilinear）两者共用同一套（`use_linear_texture_filter`，默认开），不是画质差异来源。

**关于用户提到的 `hb:a/vb:a,hqdn3d` 滤镜**：

| 问题 | 原因 |
|------|------|
| `libavfilter` 未链接 | Makefile 只有 `-lavcodec -lavformat -lavutil -lswscale -lswresample` |
| `hqdn3d` 需要 YUV 输入 | MVD 输出 ABGR32，需先转 YUV 再滤镜再转回，成本翻倍 |
| `pp=hb/vb` 需 `libpostproc` | 项目未包含 |

**可行替代**：N3DS 有余量时，可在 Morton tiling 前做一个 NEON 3×3 unsharp mask（空间锐化），不需要 libavfilter，不需历史帧，每帧约 2ms。（未实施，待用户决策）

## 4. H.265 软解线程机制分析（无代码改动）

### Codec

FFmpeg 内置 `hevc` 解码器，由 `avcodec_find_decoder(AV_CODEC_ID_HEVC)` 自动选取。

### 关键：硬编码强覆盖（decoder_video_soft.c 449-450行）

```c
// 前面 429-446 行有按 codec 能力选 FRAME/SLICE 的正常逻辑
// 但被以下两行无条件覆盖：
util_video_decoder_context[session][i]->thread_count = 2;
util_video_decoder_context[session][i]->thread_type = FF_THREAD_SLICE;
```

所有 codec 包括 H.265，**一律强制 2 线程 SLICE 模式**。`use_multi_threaded_decoding` 对线程类型无效。

### 核心分配（o3DS）

```
C0 (满载)
├── Vid_decode_thread         HIGH,  affinity=0   （调度器）
├── Vid_decode_video_thread   NORMAL,affinity=0   （调用 avcodec decode）
└── FFmpeg HEVC slice-thread-1  HIGH, fake_pthread→C0

C1 (已挤)
├── Vid_convert_thread        HIGH,  affinity=1   （YUV→RGB + Morton）
├── Vid_read_packet_thread    REALTIME,affinity=1 （读 packet）
├── Vid_audio_decode_thread   HIGH,  affinity=1   （音频解码）
└── FFmpeg HEVC slice-thread-2  HIGH, fake_pthread→C1
```

FFmpeg worker 线程由 `fake_pthread.__wrap_pthread_create` 拦截，用 3DS `threadCreate` 按 `slice_cores[]` 列表轮询分配核心。

### SLICE vs FRAME 对 H.265 的影响

| | SLICE threading（当前） | FRAME threading |
|---|---|---|
| 前提 | 编码时有多 slice header | 无要求 |
| 小分辨率 240p | 几乎所有内容 1 slice/帧 → **无并行效果** | 任意内容均有效 |
| 延迟 | 无 | 多 1 帧 |

**结论**：当前对 H.265 的硬编码 SLICE 模式在小分辨率内容上几乎等同于单线程解码，但还额外向 C1 派了一个 HIGH 优先级线程争抢资源。FRAME threading 对 H.265 效果更好（可以在 decoder_video_soft.c 按 codec_id 做区分，未实施，待用户决策）。

## 5. MVD 纹理上传三模式（上次实例已实施，本次补记）

**文件**：`source/vid_worker.c`、`source/vid_decode.c`、`include/vid_state.h`、`include/video_player.h`、`source/vid_lifecycle.c`、`source/system/sem.c`、`romfs/gfx/msg/sem_en.txt`

SEM Advanced 新增"MVD upload"三选项（仅硬解有效）：

| 选项 | 宏常量 | 实现 |
|------|--------|------|
| Classic | `VID_MVD_UPLOAD_CLASSIC` | 原始 CPU Morton tiling |
| Unroll4 | `VID_MVD_UPLOAD_UNROLL4` | 4×循环展开 + `__builtin_prefetch`，固定 Morton 偏移 |
| DMA | `VID_MVD_UPLOAD_DMA` | ABGR→linearAlloc staging buf + `GX_DisplayTransfer`（linear→tiled），`gspWaitForPPF()` 同步 |

`vid_state.h` 新增 `uint8_t mvd_upload_mode` + `uint8_t* mvd_stage[2]`（DMA 模式 staging buf）。

`vid_decode.c` 在开文件成功后按纹理尺寸 `linearAlloc` staging buf，退出时 `linearFree`。

SEM 枚举：MSG_MVD_UPLOAD=80、MSG_MVD_UPLOAD_CLS=81、MSG_MVD_UPLOAD_U4=82、MSG_MVD_UPLOAD_DMA=83，MSG_MAX 从 80 升至 84。
