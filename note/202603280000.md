## SBS 3D模式实现记录（2026-03-22，v1.62.8 已实现）

N3DS H.264硬解 SBS 3D视频播放，3DS发售10年来首次实现。
单文件H.264，MVD硬解，800×240→左右400×240，parallax barrier真3D。

视频制作命令：
```bash
ffmpeg -y -stats -stats_period 1 -i "%FILE_IN%" -filter_complex \
"[0:v]split[L][R];[L]crop=iw/2:ih:0:0[Lh];[R]crop=iw/2:ih:iw/2:0[Rh];\
[Lh]crop=min(iw\,ih*400/240):ih[Lc];[Rh]crop=min(iw\,ih*400/240):ih[Rc];\
[Lc][Rc]hstack[Vs];[Vs]format=yuv420p16le,zscale=w=800:h=240:m=bt709:min=bt709:filter=spline36,format=yuv420p[V]" \
-map "[V]" -map "0:a?" -sws_dither ed \
-c:v libx264 -preset slow -crf 15 -profile:v high -level 3.1 -fps_mode cfr \
-x264-params "aq-mode=3:aq-strength=1.3:qcomp=0.65:ref=4:bframes=4:no-fast-pskip=1" \
-color_primaries bt709 -color_trc bt709 -colorspace bt709 \
-c:a aac -b:a 128k -ac 2 -ar 48000 "%FILE_OUT%"
```

关键代码改动（video_player.c）：
| 位置 | 改动 |
|---|---|
| ~508 | 新增 `bool is_sbs_3d` 字段 |
| ~4821 | 800×240强制 is_sbs_3d=true，SAR重置1:1 |
| ~4839 | HW解码块内：is_sbs_3d=true（冗余但保留） |
| ~4884 | SBS模式EYE_LEFT纹理宽 codec_width/2，EYE_RIGHT单独初始化 |
| ~1875 | EYE_RIGHT的draw_index跟随EYE_LEFT镜像 |

## GX_DMA 实验（已回退，勿重复）

尝试用 `GX_DisplayTransfer` 替代 CPU Morton，已回退。
- 问题1：seek后花屏约2秒（H.264 seek坏帧被快速显示）
- 问题2：黑色横条（PPF事件是系统共享，无法可靠区分是谁的DMA完成）
- 结论：PPF 事件无法可靠用于 GX_DisplayTransfer 同步。

## 越播放越卡问题调查（2026-03-20，已解决）

根本原因：FFmpeg在seek时大量alloc/free小块，newlib malloc碎片化。
解决方案：seek时重开解码器（`video_player.c:5466`），close+open+seek。

- 改动：`decoder.c:Util_decoder_mvd_get_image`，移除alloc+memcpy，直接转交MVD buffer指针

### seek重开解码器（消除heap碎片）
- 改动位置：`video_player.c:5466`
- 效果：seek几十次无延迟增加，顺带解决H.264 seek花屏
