## 核心发现：citro2d z 值行为

**结论（最终确认）：z 值不可靠用于层叠，全部使用 0.5f，完全依赖绘制顺序。**

调查过程：
- citro2d `base.c` 初始化写 `C3D_DepthTest(true, GPU_GEQUAL, GPU_WRITE_ALL)`
- 理论推导"z 越大越靠前"——实验结果：**z=0.0f 的矩形反而盖住一切**（全黑屏）
- 结论：z 方向在实际设备上与理论相反，或 citro2d 内部另有覆盖，**不要用 z 做层叠控制**

**正确做法**：所有绘制 z=0.5f，靠绘制顺序（后画覆盖先画）控制层次。

## 弹窗覆盖问题（已解决）

### 问题
文件列表的文字透过错误弹窗显示。

### 根本原因
`p_draw_files_play_error_overlay()` 原先在 `draw_files_panel()` 内部调用，chrome 背景矩形也在同一函数里。弹窗虽然最后画，但 z 值错误导致不生效。

### 解决方案
1. 弹窗函数从 `draw_files_panel()` 里移出
2. 在 `Vid_panel_draw_bottom_screen()` FILES 分支**最末尾**调用（`Vid_panel_draw_overlay()` 之后）
3. 弹窗所有矩形和文字 z=0.5f，靠绘制顺序覆盖一切

### 当前弹窗结构（`p_draw_files_play_error_overlay`）
- 条件：`s_files_play_err_visible == true`
- 位置：`draw_files_panel()` 之后单独调用，绘制顺序最晚
- 内容：深蓝背景矩形 + "Unable to open file." + "Format not supported" + OK 按钮
- 输入：touch 或 A 键触发 OK，dismiss 弹窗
- hit-box：每帧绘制时更新 `s_files_play_err_ok`

## Setting 面板文字接近 chrome 消失问题

### 问题
行文字接近顶部/底部 chrome 条带时整行消失（旧代码用 `if (text_y >= VP_SET_CONTENT_Y)` 跳过整行）。

### 解决方案
移除所有手动 clamp 条件判断，chrome 矩形 z=0.5f 在内容之后画，靠绘制顺序盖住溢出内容。Setting 行内容可自由绘制，chrome 矩形视觉上覆盖超出部分。

## scissor test 尝试（失败，已撤回）

尝试用 `C3D_SetScissor` 精确裁剪内容区：
- 编译报错：函数名是 `C3D_SetScissor`，不是 `C3D_SetScissorTest`（已修）
- 坐标系问题：3DS GPU y 轴从屏幕底部向上，`top` < `bottom` 才正确
- **最终结果**：scissor 导致界面几乎全空白（背景矩形被裁掉）
- **撤回**：删除所有 `C3D_SetScissor` 调用，回归纯 z+绘制顺序方案

## draw_files_panel / draw_setting_panel 绘制顺序

```
背景矩形
→ 行内容（文字/chip/scrollbar）
→ chrome footer（盖住溢出内容）
→ chrome top（同上）
```

`Vid_panel_draw_bottom_screen` FILES 分支：
```
Draw_screen_ready
→ Vid_panel_draw_bottom()        // 含 draw_files_panel/draw_setting_panel
→ Util_expl_draw / Util_err_draw
→ Vid_panel_draw_overlay()
→ p_draw_files_play_error_overlay()  // 最后，覆盖一切
```

## 相关陷阱

- `Draw_texture_internal`（`system/draw/draw.c`）硬编码 z=0.0f，expl/err/texture 绘制均走此路径
- `Draw_c`/`Draw`/`Draw_with_background` → `Draw_internal` → `Exfont_draw_external_fonts_internal` → `Draw_texture`，文字也是 z=0.0f
- 因此所有系统绘制函数都是 z=0.0f，vid_panel 的 `C2D_DrawText/RectSolid` 用 0.5f 理论上靠前，但实际靠绘制顺序更可靠
